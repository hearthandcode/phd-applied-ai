# EXP-001 — Local-Model Curriculum Generation

## Abstract

We test whether locally-hosted open-weight language models, running on owned consumer hardware (a
10 GB-VRAM desktop GPU + a 64 GB-RAM laptop, meshed over Tailscale), can generate doctoral-level
curriculum content for the Hearth & Code PhD at a quality and **correctness** level competitive with
frontier-model (Claude) baselines — and we measure how much three escalating generation strategies
(single-model, Mixture-of-Agents, and layered Chain-of-Verification) close any gap. The motivating
constraint is correctness: curriculum a learner studies from must be *verifiably accurate*, not merely
fluent. This experiment extends the curriculum-generation methodology in `METHODOLOGY.md` to a
local-first regime and informs RQ2 (effective scaffolding configurations), RQ3 (adaptive platform
design), and RQ4 (curriculum design principles).

## 1. Background & motivation

The PhD curriculum (67 modules) is generated by an AI pipeline (`METHODOLOGY.md`): research curation →
generation → verification → cost tracking, with content calibrated to standard courses (M01 → MIT
18.06). To date this has used frontier models (Claude). A strategic pivot (hub thread T24) moved
inference onto owned hardware to (a) leverage idle compute, (b) reduce external dependence and API
cost, and (c) enable specialized local models and self-directed ML experiments. Before committing the
curriculum pipeline to local models, we must answer empirically: **can local models generate accurate,
verifiable, profile-tailored curriculum, and under what generation strategy?**

M01 (Linear Algebra) is the test bed: it already has two frontier-generated baselines (a canonical
split version under `theory/` + `foundations/`, and a `claude-generate/` monolithic version), a
calibration target (MIT 18.06, Strang), and a cognitive profile (`curriculum/cognitive-profile-v1.yaml`).

## 2. Research questions & hypotheses

**Experiment questions (methodological; map to RQ2/RQ4):**
- EQ1 — Can local models produce *structurally faithful* curriculum sections (bridges, worked examples,
  tiered memory, ML payoff, computational exercises) under the same design rules as the baselines?
- EQ2 — What is the *correctness* of single-model local output (math, code, internal consistency)?
- EQ3 — Does **Mixture-of-Agents (MoA)** improve correctness/quality over a single model?
- EQ4 — Does **layered Chain-of-Verification (CoVe)** — generate → specialized verifiers → revise —
  reduce the error rate, and by how much (especially deterministically-checkable code)?
- EQ5 — How well is the learner cognitive profile honored, and how is correctness best *guaranteed*
  (deterministic vs model-judged verification)?

**Hypotheses:**
- H1 (EQ1): Local models reproduce the section *structure* faithfully (design rules are specified, and
  these models follow format instructions well). — *expected: supported.*
- H2 (EQ2): Single-model local output contains **non-trivial correctness errors** despite high fluency
  (the "confidently wrong" failure mode). — *expected: supported.*
- H3 (EQ3): MoA improves over single-model but does **not** guarantee correctness (its error-correction
  is an aggregator side-instruction, not a dedicated check).
- H4 (EQ4): Layered CoVe with a **deterministic code verifier** reduces executable-code errors to ~zero
  and catches a majority of math/consistency errors; it is the strongest local strategy for correctness.
- H5 (EQ5): Deterministic checks (code execution, SymPy) provide hallucination-free correctness
  guarantees; model-judged verification is necessary for prose/rigor but imperfect, motivating a thin
  frontier spot-check seam.

## 3. Methodology

### 3.1 Hardware & models
- **Desktop GPU (10 GB):** `phi4-reasoning:plus` (14B, MIT — highest-rigor specialist), `deepseek-r1:14b`
  (MIT — fast drafts). **Laptop CPU (64 GB RAM):** `qwen3:30b-a3b` (MoE, Apache — 3.3B active, high
  capacity). Served via Ollama, addressed over Tailscale. Measured throughput: 7.5 / 15.4 / 9.6 tok/s
  respectively (the 30B MoE on CPU outran the 14B on GPU — see hub `model-evaluation.md`).

### 3.2 Conditions (independent variable = generation strategy)
1. **Single** — one model generates the section (primary: `phi4-reasoning:plus`; `deepseek-r1:14b` as a
   fast comparator).
2. **MoA** — proposers [phi4-reasoning:plus, qwen3:30b-a3b, deepseek-r1:14b] generate independently; an
   aggregator (phi4-reasoning:plus) synthesizes, instructed to correct cross-draft errors.
3. **CoVe (layered verification)** — a draft is passed through specialized verifiers: **code (deterministic
   execution)**, math, consistency, and pedagogical-flow (model-judged), then a reviser folds all findings
   into a corrected section; code is re-executed to confirm the fix.

### 3.3 Materials (sections)
M01 sections, spanning foundations + theory and increasing formalism density:
`foundations/01-primer`, `foundations/04-linear-independence-basis`, `theory/03-eigendecomposition`,
`theory/04-singular-value-decomposition`.

### 3.4 Controlled variables
- Same design-rule prompt + cognitive profile across all conditions (the rules the baselines used).
- **No archive-RAG** in generation (isolates model parametric knowledge; RAG/citations are a separate
  later variable). This is a deliberate limitation — see §7.

### 3.5 Grading rubric (per section × condition)
Aligned with `VERIFICATION-RUBRIC.md` (LORI 1.5 + Achieve OER + OERTrust), with correctness elevated to
the primary axis given the research priority on verifiable accuracy:
- **Correctness (0–5)** — every definition/derivation correct; code executes; no false claims. *(partly
  deterministic: code execution + SymPy where applicable.)*
- **Rigor (0–5)** — preconditions stated; complete definitions.
- **Pedagogy (0–5)** — bridges, worked examples, tiered memory, ADHD-aware design (per the baselines).
- **Profile-fit (0–5)** — tone/pacing vs `cognitive-profile-v1.yaml`.
Baselines (canonical + claude-generate) are graded on the same rubric as the reference ceiling.

### 3.6 Procedure & reproducibility
Generation: `artifacts/gen_m01_local.py`. Deterministic code check: `artifacts/verify_code.py`. Layered
verification pipeline: `artifacts/verify_pipeline.py`. Raw outputs in `data/`; grades in
`analysis/gap-analysis.md`. All runs are over the Tailscale mesh against the two nodes; see `data/run.log`.

## 5. Findings (preliminary)

- **F1 (supports H1):** Local single-model output is *structurally* faithful — bridges, formal
  definitions, worked examples, tiered CARRY/RECONSTRUCT/LOOKUP, ML payoff, NumPy exercise, and
  explain-out-loud check were all present in the deepseek primer.
- **F2 (supports H2):** It also contained **real, confidently-presented errors**: an executable exercise
  that crashes; a false claim that a 2×2 matrix represents a ℝ²→ℝ³ map; a vector-space definition reduced
  to closure only (omitting the 8 axioms); minor terminological imprecision. **Fluency ≠ correctness.**
- **F3 (supports H4/H5):** A **deterministic code verifier** (execute every code block) caught the
  crashing exercise instantly with zero hallucination risk — the strongest correctness guarantee
  available, requiring no model.
- F4 (pending): whether phi4-reasoning:plus avoids these errors; whether MoA self-corrects them; whether
  the CoVe pipeline repairs them (target: code 1→0, math/consistency flags raised and fixed).

## 6. Implications for the research

- **RQ2 (effective scaffolding configurations):** The data indicate the *effective* configuration for
  local curriculum generation is not a single model but a **pipeline**: generate (ensemble) → verify
  (deterministic-first, then model-judged, diverse) → revise, with a thin frontier seam for the residual.
  Configuration, not raw model power, is the lever.
- **RQ3 (adaptive platform design):** A local-first generation+verification stack is viable on consumer
  hardware ($0 inference), strengthening the case that the H&C platform's content pipeline can be
  self-hosted and privacy-preserving, with frontier APIs as an optional augmentation rather than a
  dependency.
- **RQ4 (curriculum design principles):** Reinforces a design principle already implicit in the
  baselines — **content must be verifiable, and the cheapest reliable verification is executable**
  (runnable code, symbolic math checks). Curriculum should be authored so that as many claims as possible
  are mechanically checkable. This is a generalizable principle for AI-generated educational content.
- **Methodology extension:** `METHODOLOGY.md` Phase 3 (Verification) should be elevated from a rubric pass
  to an explicit **layered verification stage** (deterministic + diverse model verifiers + spot-check),
  and Model Routing extended to the two-node local farm + hybrid frontier seam.

## 7. Limitations

- **No-RAG control:** generation used parametric knowledge only; citation accuracy and gap-filling from
  external sources (archive MCP + web) are deferred to a follow-up (EXP-002). Real pipeline output will
  be retrieval-augmented and is expected to be more accurate.
- **Single test domain (linear algebra):** well-represented in pretraining; correctness on lower-resource
  topics (frontier ML, niche CS) may differ and needs separate testing.
- **Model-judged verifier fallibility:** math/consistency/flow verifiers can miss or false-flag;
  mitigated by deterministic checks, verifier diversity, and frontier spot-checks, but not eliminated.
- **n=1 sections per condition initially:** broaden before drawing strong quantitative conclusions.
